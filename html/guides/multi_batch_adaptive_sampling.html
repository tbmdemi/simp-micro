What & Why

A self-contained sub-pipeline that replaces one-shot design with iterative, coverage-driven exploration.

Một quy trình con độc lập, thay thế thiết kế một-lần bằng khám phá lặp theo vùng phủ.

Phase 1 screens a coarse grid of seed × objective × parameter combos. It tells you what works — but it cannot tell you what you missed. The multi-batch adaptive pipeline solves this by operating in sequential batches:

Pha 1 sàng lọc một lưới thô các tổ hợp seed × objective × tham số. Nó cho bạn biết cái gì hiệu quả — nhưng không thể cho biết bạn đã bỏ sót gì. Quy trình thích ứng đa batch giải quyết điều này bằng cách vận hành theo các batch tuần tự:

Batch 1 — Sobol / LHS space-filling design across the full parameter domain (120 samples).
Batch 2+ — Coverage analysis identifies under-explored regions; decide_next_action() chooses to expand, refine, or stop.

Each batch runs SIMP in parallel (4 workers), aggregates results, generates self-contained HTML coverage reports, and logs every decision. The loop continues until the property space is adequately sampled or improvement plateaus.

Mỗi batch chạy SIMP song song (4 worker), tổng hợp kết quả, tạo báo cáo vùng phủ HTML độc lập, và ghi nhật ký mọi quyết định. Vòng lặp tiếp tục cho đến khi không gian thuộc tính được lấy mẫu đầy đủ hoặc cải thiện đạt đến ngưỡng bão hòa.

Sampling Strategies / Chiến lược lấy mẫu

Three space-filling designs, each with distinct trade-offs.

Ba thiết kế lấp đầy không gian, mỗi loại có sự đánh đổi riêng.

Sobol Deterministic Low-Discrepancy / Độ lệch thấp xác định

Uses scipy.stats.qmc.Sobol to generate a sequence that covers the hypercube uniformly. Every run with the same seed produces identical designs — critical for reproducibility.

Sử dụng scipy.stats.qmc.Sobol để tạo một chuỗi bao phủ siêu khối lập phương đồng đều. Mỗi lần chạy với cùng seed tạo ra các thiết kế giống hệt nhau — rất quan trọng cho khả năng tái lập.

from scipy.stats.qmc import Sobol
sampler = Sobol(d=n_params, scramble=True, seed=42)
sample = sampler.random(n=n_samples)
# Scale to param ranges → DataFrame with seed/objective columns
  • Best for / Tốt nhất cho — Batch 1 initial exploration, when you want deterministic results / khám phá ban đầu batch 1, khi cần kết quả xác định
  • Drawback / Hạn chế — Can align with periodic structure in some objectives / có thể căn chỉnh với cấu trúc tuần hoàn trong một số mục tiêu
  • Default / Mặc định — Yes, used as the default strategy / có, được dùng làm chiến lược mặc định

LHS Latin Hypercube Sampling

Stratified random sampling via pyDOE.lhs (or a built-in fallback). Divides each dimension into n equally-probable intervals and places one sample per interval.

Lấy mẫu ngẫu nhiên phân tầng qua pyDOE.lhs (hoặc fallback tích hợp sẵn). Chia mỗi chiều thành n khoảng có xác suất bằng nhau và đặt một mẫu vào mỗi khoảng.

from pyDOE import lhs
design = lhs(n=n_params, samples=n_samples, criterion='maximin')
# Randomised, non-deterministic by default / Ngẫu nhiên, không xác định theo mặc định
  • Best for / Tốt nhất cho — Stochastic robustness testing, when you want randomness / kiểm tra độ bền vững ngẫu nhiên, khi bạn muốn tính ngẫu nhiên
  • Drawback / Hạn chế — Can leave larger gaps than Sobol in low dimensions / có thể để lại khoảng trống lớn hơn Sobol ở chiều thấp
  • pyDOE required / Yêu cầu pyDOE — Optional dependency; fallback uses numpy.random.uniform / phụ thuộc tùy chọn; fallback dùng numpy.random.uniform

Optimized LHS Correlation-Minimized LHS / LHS tối thiểu tương quan

Generates multiple LHS candidates and selects the one with minimum pairwise column correlation (SPSA-like heuristic). Produces better space-filling at the cost of computation.

Tạo nhiều ứng viên LHS và chọn ứng viên có tương quan cột tối thiểu (heuristic dạng SPSA). Tạo ra sự lấp đầy không gian tốt hơn nhưng tốn nhiều chi phí tính toán hơn.

def optimized_lhs(n_params, n_samples, n_trials=20):
    best_corr = 1.0
    for _ in range(n_trials):
        candidate = lhs(n_params, n_samples)
        corr = np.mean(np.abs(np.corrcoef(candidate, rowvar=False)))
        if corr < best_corr: best = candidate
    return best
  • Best for / Tốt nhất cho — High-dimensional parameter spaces (>5 params) / không gian tham số nhiều chiều (>5 tham số)
  • Drawback / Hạn chế — Slower to generate, not deterministic / chậm hơn để tạo, không xác định
  • Used when / Dùng khi — --strategy optimized_lhs

Design output format: A pandas.DataFrame with columns for each active parameter (volfrac, void_size_frac, …) plus seed (shape name) and objective (auxetic/first/second). This is the contract that runner.py expects.

Coverage Analysis / Phân tích vùng phủ

How we measure design space exploration completeness.

Cách chúng tôi đo lường mức độ khám phá không gian thiết kế.

The coverage engine operates on the property space — not the parameter space. It discretises v12, v21, and obj_value into an ND grid and counts how many bins contain at least one sample.

Cơ chế vùng phủ hoạt động trên không gian thuộc tính — không phải không gian tham số. Nó lượng tử hóa v12, v21 và obj_value thành một lưới đa chiều và đếm số lượng bin có ít nhất một mẫu.

def coverage_report(results, dims=('v12', 'v21', 'obj_value'), n_bins=10):
    """Bin each property dimension → count samples per bin → report sparsity."""
    grid = np.zeros([n_bins] * len(dims))
    for r in results:
        coords = tuple(np.digitize(r[d], bins[i]) for i, d in enumerate(dims))
        grid[coords] += 1
    sparsity = np.mean(grid == 0)
    return {'coverage_pct': (1 - sparsity) * 100, 'grid': grid}
  • coverage_report() — Returns coverage percentage, per-bin counts, and sparsity statistics / trả về tỷ lệ vùng phủ, số lượng mỗi bin và thống kê độ thưa
  • find_sparse_regions() — Returns list of bin coordinates with < N samples (configurable threshold) / trả về danh sách tọa độ bin có < N mẫu (ngưỡng cấu hình được)
  • compare_batches() — Reports how many new bins each batch added (coverage improvement) / báo cáo số lượng bin mới mỗi batch thêm vào (cải thiện vùng phủ)

Decision Logic / Logic quyết định

Three possible actions, determined by coverage metrics and convergence history.

Ba hành động có thể, được xác định bởi các chỉ số vùng phủ và lịch sử hội tụ.

🛑 Stop / Dừng

condition: no_improvement ≥ 2 batches

Objective has not improved for 2+ consecutive batches and coverage is adequate (< 15% sparsity). Time to consolidate and report.

Mục tiêu không cải thiện sau 2+ batch liên tiếp và vùng phủ đầy đủ (độ thưa < 15%). Đã đến lúc tổng hợp và báo cáo.

🚀 Expand / Mở rộng

condition: sparsity > 30%

Large regions of the property space remain unexplored. The next batch adds new seed shapes or objectives, or uses a broader sampling strategy.

Vùng rộng lớn của không gian thuộc tính vẫn chưa được khám phá. Batch tiếp theo thêm các shape seed hoặc objective mới, hoặc sử dụng chiến lược lấy mẫu rộng hơn.

🎯 Refine / Tinh chỉnh

condition: sparsity < 10% & best_obj far

The property space is well-covered but the best objective is still far from theoretical limits. The next batch narrows parameter bounds around the best-known region and uses a denser local design.

Không gian thuộc tính được phủ tốt nhưng objective tốt nhất vẫn còn xa giới hạn lý thuyết. Batch tiếp theo thu hẹp ranh giới tham số quanh vùng được biết đến tốt nhất và sử dụng thiết kế cục bộ dày đặc hơn.

Decision thresholds are configurable: sparsity_expand_threshold, sparsity_refine_threshold, improvement_patience. Tune them via the decide_next_action() function signature — no subclassing needed.

Why This Way? / Tại sao làm theo cách này?

Technical decisions that shaped the module.

Các quyết định kỹ thuật đã định hình module.

Stateless persistence / Bền vững không trạng thái

Every batch writes its summary, results, params, and design as separate JSON/CSV files. No database, no global state — just a directory tree you can inspect, diff, and resume from.

Mỗi batch ghi summary, kết quả, tham số và thiết kế thành các file JSON/CSV riêng biệt. Không database, không trạng thái toàn cục — chỉ một cây thư mục bạn có thể kiểm tra, so sánh và tiếp tục.

Mock-mode for development / Chế độ giả lập cho phát triển

--skip-run generates realistic mock results (uniform random in property space with ~10% failure rate). This lets you test the full decision loop without running SIMP.

--skip-run tạo kết quả giả thực tế (ngẫu nhiên đồng đều trong không gian thuộc tính với tỷ lệ thất bại ~10%). Điều này cho phép bạn kiểm tra toàn bộ vòng lặp quyết định mà không cần chạy SIMP.

Zero new dependencies / Không phụ thuộc mới

The module uses only scipy, numpy, pandas — all already in requirements.txt. pyDOE for LHS is an optional import; a pure-numpy fallback is provided.

Module chỉ sử dụng scipy, numpy, pandas — tất cả đã có trong requirements.txt. pyDOE cho LHS là import tùy chọn; fallback bằng numpy thuần được cung cấp.

Property-space coverage / Vùng phủ không gian thuộc tính

We analyze coverage in property space (v12, v21, obj) — not parameter space. This is deliberate: it tells you if you're discovering genuinely different designs, not just different inputs.

Chúng tôi phân tích vùng phủ trong không gian thuộc tính (v12, v21, obj) — không phải không gian tham số. Điều này có chủ ý: nó cho biết bạn có đang khám phá các thiết kế thực sự khác nhau hay không, chứ không chỉ các đầu vào khác nhau.

Configurable decision thresholds / Ngưỡng quyết định cấu hình được

All thresholds in decide_next_action() are keyword arguments with sensible defaults. No configuration file, no magic numbers — just Python that you can tune per experiment.

Tất cả ngưỡng trong decide_next_action() là đối số keyword với giá trị mặc định hợp lý. Không file cấu hình, không số ma thuật — chỉ Python mà bạn có thể tùy chỉnh cho từng thí nghiệm.

Self-contained HTML reports / Báo cáo HTML độc lập

Coverage reports and batch progression visualisations are single-file HTML pages — no server needed. Generated at every batch so you can monitor progress in real time.

Báo cáo vùng phủ và trực quan hóa tiến trình batch là các trang HTML đơn file — không cần server. Được tạo ở mỗi batch để bạn có thể theo dõi tiến trình theo thời gian thực.

Integration: Phase 1 to Multi-Batch / Tích hợp: Pha 1 vào Multi-Batch

The multi-batch pipeline reads from refined_parameters.json (produced by pipeline/params.py):

Pipeline đa batch đọc từ refined_parameters.json (được tạo bởi pipeline/params.py):

{
  "active_parameters": {
    "volfrac": { "range": [0.3, 0.7] },
    "void_size_frac": { "range": [0.1, 0.4] }
  },
  "fixed_parameters": {
    "rmin": 1.5,
    "move": 0.2,
    "penal": 3.0
  },
  "multi_batch": {
    "sampling_strategies": ["sobol", "lhs", "optimized_lhs"],
    "n_batch1_default": 120,
    "max_batches_default": 5,
    "decision_thresholds": {
      "sparsity_expand": 0.30,
      "sparsity_refine": 0.10,
      "improvement_patience": 2
    }
  }
}

No migration needed. If you have a Phase 1 summary with active_parameters and ranges, it works. The multi_batch key in REFINED_PARAMETERS_TEMPLATE is a convenience addition — the pipeline can also parse plain parameters dicts from any JSON summary file.
